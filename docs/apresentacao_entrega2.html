Recap — Entrega 1
Problema e Motivação
Por que combinar LLMs e verificação formal?

O que LLMs fazem bem

  • Identificam padrões suspeitos no código
  • Analisam semântica e contexto
  • Rápidos e escaláveis

Problema dos LLMs

  • Geram alucinações e falsos positivos
  • Confundem bugs executáveis com code smells
  • Resultados variam por modelo e prompt
  • Sem garantia formal de correção

O que o ESBMC oferece

  • Verificação formal e determinística
  • Contraexemplos concretos para bugs
  • Sem falsos positivos para bugs verificáveis
  • F1 = 1.00 para as categorias suportadas

Ideia central: usar a LLM como guia inteligente para o ESBMC, e o ESBMC como filtro formal das suspeitas da LLM

Pesquisa
Pergunta de Pesquisa e Objetivo

Problema central

  • LLMs produzem resultados diferentes dependendo do modelo e do prompt
  • Bugs e code smells têm características estruturais semelhantes — difíceis de diferenciar
  • Nem toda suspeita levantada pela LLM corresponde a uma falha real executável

Como combinar LLMs e verificação formal para aumentar a confiabilidade da análise automática de código Python?

Objetivo geral

Desenvolver e avaliar um pipeline híbrido que combine LLMs e ESBMC-Python para análise automática de funções Python, medindo o impacto de cada componente sobre a qualidade dos resultados.

Design experimental
Três Fluxos Experimentais
Todos comparados contra o mesmo ground truth

Flow A — ESBMC-only

Baseline Formal

O código Python é enviado diretamente ao ESBMC, sem participação de LLM. Serve como teto superior para bugs formais — determinístico, sem falsos positivos, mas cego a code smells.

Flow B — Híbrido (LLM + ESBMC)

Abordagem Principal

LLM analisa a função e levanta hipóteses → AST valida as referências estruturais → findings verificáveis são enviados ao ESBMC → ESBMC confirma ou rejeita cada hipótese formalmente.

Flow C — LLM-only

Baseline Neural

A LLM analisa a função sem validação formal. Avalia tanto bugs quanto code smells. Mede o custo de não ter verificação — o quanto de ruído a LLM sozinha produz.

💬 Perguntas que cada fluxo responde: A → "ESBMC sozinho é suficiente?", B → "A LLM melhora o ESBMC? O ESBMC filtra a LLM?", C → "Qual é o custo de usar só LLM?"
Flow B — Detalhamento
Pipeline Híbrido Passo a Passo
1

Entrada: CodeUnit

Código-fonte da função + assinatura + metadados mínimos (sem categoria, sem ground truth)

2

LLM analisa via Prompt de 8 passos

Modelo recebe a função e produz hipóteses: tipo de problema, linha suspeita, justificativa

3

AST valida as referências

O Python AST verifica se as linhas/variáveis citadas pela LLM realmente existem na estrutura do código

4

Triagem: verificável vs. não verificável

Findings formais (division_by_zero, out_of_bounds, assertion_violation) → ESBMC. Code smells → classificados diretamente.

5

ESBMC confirma ou rejeita

Gera contraexemplo concreto (FALHA) ou prova de segurança (SAFE) para cada hipótese verificável

Categorias de saída
Bug verificável — LLM apontou + ESBMC confirmou com contraexemplo
Code smell — LLM identificou problema estrutural não verificável pelo ESBMC
Falso positivo — LLM apontou bug, mas ESBMC retornou SAFE
Out-of-scope — Finding não pode ser verificado formalmente nem classificado como smell padrão
💬 O NRR (Noise Reduction Rate) mede exatamente quantos falsos positivos da LLM o ESBMC consegue eliminar nessa triagem.
Metodologia
Prompt da LLM — Design Cuidadoso
O que a LLM vê e o que ela NÃO vê

✅ Fornecido à LLM

  • Código-fonte completo da função
  • Assinatura (nome, parâmetros, tipos)
  • Metadados mínimos (ex: nome do arquivo)
Prompt estruturado em 8 passos orienta a LLM a raciocinar sistematicamente: identificar variáveis críticas, condições perigosas, linha suspeita, tipo de bug e justificativa.

🚫 Ocultado da LLM

  • Categoria do dataset (ex: "division_by_zero") — evita viés por rótulo
  • Ground truth — a LLM não sabe se a função tem bug
  • Operações AST pré-extraídas — análise parte do zero
Por quê? Simular análise cega e realista — como seria em produção, sem saber de antemão o que procurar.

Impacto do design do prompt: O mesmo código analisado com prompts diferentes produz resultados diferentes. O prompt de 8 passos foi desenvolvido para forçar raciocínio estruturado antes da conclusão final — reduzindo alucinações superficiais.

Avaliação
Métricas de Avaliação
Finding-level + Function-level + Híbridas

📐 Precision

TP / (TP + FP)

Dos bugs apontados, quantos estavam corretos? Alta precision = poucos alarmes falsos.

🔍 Recall

TP / (TP + FN)

Dos bugs reais existentes, quantos foram encontrados? Alto recall = poucos bugs perdidos.

⚖️ F1-Score

2 × P × R / (P + R)

Média harmônica entre Precision e Recall. Equilíbrio entre não errar e não perder.

🎯 MCC

Matthews Corr. Coef.

Considera TP, FP, FN e TN. Melhor que Accuracy quando o dataset é desbalanceado (o nosso é).

✅ FCR

confirmados / tentativas

Formal Confirmation Rate. Mede a qualidade das hipóteses da LLM — quantas sobrevivem ao ESBMC.

🔕 NRR

(FP_LLM − FP_Hybrid) / FP_LLM

Noise Reduction Rate. Mede quanto ruído (FP) o ESBMC elimina em relação ao LLM puro.

FCR alto = LLM produz hipóteses precisas. NRR alto = ESBMC elimina muito ruído da LLM. Os dois juntos mostram o valor real do pipeline híbrido.

Análise
O que os Resultados nos Dizem

Validação formal sempre ajuda

  • Precision aumentou em todos os modelos após ESBMC
  • DeepSeek: 0.75 → 0.96
  • Qwen: 0.61 → 0.97
  • GPT-4o: 0.80 → 0.92

Recall se mantém após ESBMC

  • Recall não cai: o ESBMC só filtra FPs, não perde TPs
  • GPT-4o mantém Recall=0.98 no híbrido
  • Isso é crucial: nenhum bug real é perdido na filtragem

Alucinações estruturais: 21 total

  • DeepSeek: 7 alucinações (21.9% dos findings)
  • Qwen: 13 alucinações (24.5%) — mais propenso
  • GPT-4o: apenas 1 (1.8%) — bem calibrado
  • O AST elimina essas antes de chegar ao ESBMC

Smells são mais difíceis

  • Sem validação formal, qualidade depende muito do modelo
  • Qwen: Recall=1.00 mas Precision=0.13 (inútil em produção)
  • DeepSeek: F1=0.30 — não confiável para smells
  • Apenas GPT-4o tem desempenho aceitável (F1=0.81)

Limitações da V1

Limitações do Dataset

  • Apenas 70 funções — pequeno para conclusões robustas
  • Construído manualmente — viés de construção
  • Apenas 3 categorias de bugs formais
  • Apenas 3 categorias de code smells
  • Não avalia projetos Python completos — só funções isoladas

Limitações dos Modelos

  • Resultados variam por modelo — não generalizáveis
  • Mesmo modelo pode produzir respostas diferentes para o mesmo código (temperatura)
  • Modelos locais (7b) limitados em capacidade de raciocínio

Limitações do ESBMC-Python

  • Só verifica código Python simples sem dependências externas
  • Não escala para projetos reais com libs de terceiros
  • F1=1.00 no benchmark é artefato da construção manual do dataset

Essas limitações são conhecidas e documentadas — fazem parte do design V1. A V2 foi planejada para endereçar cada uma delas.
