# V1 — Escopo do artigo

Esta versão é o foco da dissertação. O objetivo é demonstrar que LLMs podem orientar o ESBMC a verificar propriedades em código Python que o BMC sozinho não alcançaria.

## Fluxo principal

```
arquivo Python → LLM → AST → Formalizer → ESBMC → Relatório → Ground Truth
```

1. **Preprocess (AST):** extrai funções, operações, métricas de complexidade.
2. **LLM Analyzer:** analisa o código e retorna achados em JSON (bugs suspeitos + smells).
3. **AST Validation:** filtra alucinações — se a expressão não existe no AST executável, é `llm_false_positive`.
4. **Formalizer:** converte achados verificáveis em propriedades formais (assertion + assumptions + flags ESBMC).
5. **Instrumenter:** injeta a propriedade no código como driver simbólico.
6. **ESBMC:** roda verificação formal por BMC no arquivo instrumentado.
7. **Report:** classifica o resultado final de cada achado.
8. **Ground Truth:** compara com anotações manuais e calcula P/R/F1.

## Categorias

### Bugs formais (verificáveis pelo ESBMC)

| Categoria | Exceção |
|---|---|
| `division_by_zero` | `ZeroDivisionError` |
| `out_of_bounds` | `IndexError` |
| `assertion_violation` | `AssertionError` |

### Code smells (heurísticos — secundários)

| Categoria |
|---|
| `long_method` |
| `many_parameters` |
| `complex_conditional` |

## Classificações principais

| Classificação | Descrição | Papel nas métricas |
|---|---|---|
| `llm_confirmed_by_esbmc` | ESBMC confirmou formalmente | **TP principal** |
| `not_confirmed_within_bound` | ESBMC verificou, sem violação no bound | FP ou limitação |
| `esbmc_inconclusive` | Erro, timeout ou ambiguidade | excluir do F1 |
| `llm_false_positive` | Alucinação detectada pelo AST | FP da LLM |
| `heuristic_smell_only` | Smell heurístico | métrica separada |
| `skipped_not_verifiable` | Formalizer não gerou propriedade real | registrar, excluir do F1 formal |
| `esbmc_native_bug` | ESBMC direto achou sem LLM | baseline comparison |
| `llm_missed_esbmc_bug` | ESBMC achou, LLM não reportou | FN da LLM |

## Modos de execução

```bash
# Pipeline completo (recomendado)
python src/main.py --mode full \
  --input dataset/labeled/ok/bugs \
  --model gpt-4o \
  --bound 5 --timeout 30 \
  --report reports/json/full_v1_validation.json

# Benchmark com ground truth
python src/main.py --mode benchmark \
  --input dataset/labeled/ground_truths/bugs \
  --model gpt-4o \
  --bound 5 --timeout 30 \
  --report reports/json/benchmark_v1.json

# Baseline ESBMC sem LLM
python src/main.py --mode esbmc-direct \
  --input dataset/labeled/ok/bugs \
  --bound 5 --timeout 30
```

## Métricas para o artigo

O modo `benchmark --report` gera:

```json
{
  "metrics": {
    "bugs_llm_only":         { "precision", "recall", "f1", "tp", "fp", "fn" },
    "bugs_hybrid_pipeline":  { "precision", "recall", "f1", "tp", "fp", "fn" },
    "smells":                { "precision", "recall", "f1", "tp", "fp", "fn" },
    "esbmc_direct_baseline": { "precision", "recall", "f1", "tp", "fp", "fn" }
  },
  "hallucinations": { "count", "rate" }
}
```

**Regras:**
- Calcular F1 **separadamente** para bugs formais e smells.
- `bugs_llm_only` mede achados da LLM depois da validação AST.
- `bugs_hybrid_pipeline` mede apenas achados confirmados pela trilha formal LLM + Formalizer + Instrumenter + ESBMC.
- Não misturar `llm_confirmed_by_esbmc` com resultados de harness.
- Harness runtime fica fora das métricas principais da V1.
- Casos `clean` são controles negativos: ausência de achados é comportamento correto; qualquer bug/smell reportado conta como falso positivo.
- Quando o Formalizer não consegue extrair denominador, base, índice ou condição real, o achado deve virar `skipped_not_verifiable`, nunca uma confirmação com propriedade artificial.
- `esbmc_direct_baseline` serve para comparar com a trilha LLM+ESBMC.
